## C.3 FoodLookup — performance

Arquivo: `backend/app/services/ai/food_lookup.py` (173 linhas).

**N-gramas gerados (medido via `_extract_candidates`)**

| Texto típico | Palavras | N-gramas |
|---|---|---|
| "200g de arroz branco cozido com 1 bife de frango grelhado e brócolis" | 13 | **33** |
| "café com leite e 2 fatias de pão integral com manteiga" | 11 | **27** |
| "salada de alface tomate cebola pepino com azeite e atum" | 10 | **24** |
| "200g arroz" | 2 | 1 |

**EXPLAIN ANALYZE** (artefato `C3-food-lookup-explain.txt`, banco com 42103 alimentos)

| Variante da query | Tempo | Plano |
|---|---|---|
| Atual: `%>> :q OR similarity() >= 0.18` | **585 ms** | **Seq Scan** — índice GIN não usado |
| Só `%>> :q` | **8 ms** | Bitmap Index Scan em `ix_foods_search_trgm` |
| Só `similarity() >= 0.18` | 249 ms | Seq Scan |

**Diagnóstico** — o `OR similarity() >= 0.18` é fallback para alimentos com baixa similaridade que `%>>` (com threshold default `pg_trgm.set_limit = 0.3`) não encontra. Mas o predicado funcional `similarity() >= X` não é indexável, e o planner do Postgres descarta o uso do índice em todo o `WHERE` por causa do `OR`. Resultado: cada candidato leva ~600ms em vez de ~10ms (**~70× mais lento**).

**Impacto cumulativo combinado com AUD-006 (N+1)**

| Cenário | Por candidato | × 25 candidatos | Bloqueia worker uvicorn por |
|---|---|---|---|
| Hoje (Seq Scan + N+1) | 585 ms | **14.6 s** | 14.6 s |
| Só corrigir índice (sem fix N+1) | 8 ms | 200 ms | 200 ms |
| Só corrigir N+1 (mantém Seq Scan, 1 query batch) | ~600 ms | ~600 ms | ~600 ms |
| Os dois | ~10 ms | ~10 ms | ~10 ms |

**Recomendação**

1. **Imediato**: trocar OR por configuração de threshold (`SET pg_trgm.similarity_threshold = 0.18` por sessão antes da query, e usar só `WHERE search_text %>> :q`). O `%>>` respeita o threshold dinâmico e mantém o índice.
2. **Estrutural**: substituir trigrama por `tsvector` (FTS) com `ts_rank_cd` — melhor para busca multi-palavra, paramento de stop-words em português, e indexável.
3. **Combinado com AUD-006**: bater os 25 candidatos numa única query batch (`UNION ALL` ou `WHERE search_text %>> ANY(:array)`).

Achado registrado: AUD-016 (🔴 crítica) — query causa Seq Scan no caminho síncrono do usuário.
